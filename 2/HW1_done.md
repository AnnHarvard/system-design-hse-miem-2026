# HW1 — Architectural Kata

## Тема

### We're Taking Requests...


Местная радиостанция хочет укрепить связь между диджеями и слушателями, чтобы слушатели могли заказывать песни, голосовать за композиции, которые звучат в эфире, голосовать за ежедневный плейлист диджея и т. д.

Пользователи: пока не определено; это будет местное «музыкальное» сообщество

Требования: синхронизация с музыкой в эфире практически в режиме реального времени; механизм голосования пользователей; доступность с мобильных устройств

---

---

### Риски проекта

| Риск | Возможная проблема |
|---|---|
| Большое число пользователей одновременно | Система может начать медленно обновлять данные |
| Повторные голосования или спам | Искажение результатов голосования |
| Требование быстрого обновления информации | Необходимо обеспечить обновление данных почти без задержек |
| Ограниченный бюджет | Нельзя использовать слишком дорогие технологии и серверы |

---

### Критерии успешности MVP

Минимальная версия системы считается успешной, если пользователь может:

- открыть приложение с телефона;
- увидеть текущую песню в эфире;
- проголосовать за трек;
- отправить запрос на песню;
- получать обновления эфира почти в реальном времени.

- отображение информации об эфире.

Система не отвечает за:
- хранение музыкальных файлов;
- аудиостриминг;
- обработку авторских прав.

### Основные пользовательские сценарии

#### Сценарий 1 — Просмотр текущей песни и голосование
1. Пользователь открывает приложение на телефоне.
2. Система показывает, какая песня сейчас играет.
3. Пользователь нажимает кнопку голосования.
4. Система сохраняет голос.
5. Обновленный результат становится доступен другим пользователям почти сразу.

---

#### Сценарий 2 — Отправка запроса на песню
1. Пользователь открывает форму запроса.
2. Вводит название песни или исполнителя.
3. Отправляет запрос.
4. Система сохраняет запрос.
5. DJ видит его в списке новых запросов.

---

#### Сценарий 3 — Работа DJ с плейлистом
1. DJ открывает панель управления.
2. Видит текущий плейлист и список запросов слушателей.
3. Может добавить или убрать песню из плейлиста.
4. Система обновляет данные для всех пользователей.

## 1.4 Нефункциональные требования

### Цель этапа

Зафиксировать технические характеристики системы, которые будут влиять на архитектуру, выбор хранилища, способ обновления данных и способы защиты от перегрузки.

---

### 1. Производительность

| Показатель | Требование | Почему это важно |
|---|---|---|
| Время ответа API на голосование | не более 300 мс в p95 | Пользователь должен видеть, что голос засчитан почти сразу |
| Время ответа API на отправку запроса песни | не более 300 мс в p95 | Запрос не должен “висеть” в интерфейсе |
| Задержка обновления текущего трека | не более 2 секунд | Состояние эфира должно быть почти синхронным у всех пользователей |
| Задержка обновления счетчика голосов | не более 1–2 секунд | Голосование должно визуально ощущаться как живое |
| Открытие основной страницы на мобильном устройстве | не более 2 секунд при обычной сети | Пользовательский сценарий должен быть быстрым |

Требование по задержкам означает, что обычный polling не должен быть основным способом обновления данных. Для доставки изменений лучше подходит WebSocket или SSE, а polling может использоваться только как запасной вариант.

---

### 2. Нагрузка

Для MVP можно принять следующие ориентиры:

| Параметр | Оценка |
|---|---|
| DAU | около 10 000 |
| Пиковое число одновременно активных пользователей | около 2 000 |
| Голосований в день | около 100 000 |
| Пиковая нагрузка на запись | 20–50 RPS |
| Пиковая нагрузка на чтение текущего состояния | 300–500 RPS |

Особенности нагрузки:
- система будет read-heavy, потому что большинство пользователей в основном смотрят текущий трек и результаты;
- запись голосов и запросов происходит реже, но должна быть надежной;
- пики нагрузки возможны в момент смены песни, во время популярного трека или после анонса DJ.

Из этого следует, что архитектура должна отдельно учитывать:
- быстрый ответ на чтение текущего состояния;
- надежную запись голосов;
- отдельный канал для realtime-обновлений.

---

### 3. Масштабируемость

Система должна масштабироваться без полной переделки архитектуры.

| Область | Требование |
|---|---|
| API | backend должен быть stateless |
| Realtime-обновления | должны поддерживать несколько серверов одновременно |
| Хранилище | должно выдерживать рост количества голосов и запросов |
| Кэш | должен разгружать базу при частых чтениях |

Практически это означает:
- несколько экземпляров backend за балансировщиком нагрузки;
- отдельный слой для публикации событий об обновлении эфира;
- хранение часто читаемых данных в кэше;
- возможность увеличивать нагрузку горизонтально, а не только “более мощным сервером”.

---

### 4. Надежность и сохранность данных

| Показатель | Требование |
|---|---|
| Потеря голоса | недопустима |
| Потеря запроса на песню | недопустима |
| Повторная отправка одного и того же действия | не должна искажать результат |
| Сбой одного backend-инстанса | не должен останавливать систему |

Требования к поведению:
- голос должен записываться атомарно;
- повторный запрос от пользователя не должен считаться как новый голос;
- после временного сбоя данные должны сохраняться в постоянном хранилище;
- результаты должны сходиться после синхронизации, даже если разные пользователи видят обновление с небольшой задержкой.

---

### 5. Согласованность данных

Для этой задачи допустима не строгая, а практически полезная согласованность.

| Операция | Требование |
|---|---|
| Учет голоса | атомарная запись |
| Отображение счетчика голосов | допустима небольшая задержка обновления |
| Смена текущего трека | должна дойти до пользователей почти сразу |
| Список запросов | может обновляться с небольшой задержкой |

Это означает, что:
- для записи голосов нужна надежная транзакционная логика или идемпотентная обработка;
- для отображения результатов допускается eventual consistency;
- клиент не обязан видеть абсолютно одно и то же значение в одну и ту же миллисекунду, но расхождение не должно быть заметным для пользователя.

---

### 6. Отказоустойчивость

| Ситуация | Требование |
|---|---|
| Падение одного сервиса | система продолжает работать |
| Разрыв соединения у клиента | клиент должен переподключаться автоматически |
| Недоступность realtime-канала | показывается последнее известное состояние |
| Краткий сбой базы данных | должны быть предусмотрены повторные попытки |

Требования к архитектуре:
- не должно быть единственной точки отказа там, где это критично;
- realtime-клиент должен уметь переподключаться;
- при временной проблеме backend должен возвращать понятную ошибку и не терять данные;
- состояние эфира должно быть восстановимо после сбоя.

---

### 7. Безопасность

| Показатель | Требование |
|---|---|
| Передача данных | только по HTTPS |
| Повторное голосование | должно быть ограничено |
| Спам-запросы | должны отсекаться rate limiting-ом |
| Валидация входных данных | обязательна на сервере |

Для защиты системы нужны:
- идентификатор пользователя или устройства для ограничения повторных голосов;
- ограничение числа запросов с одного пользователя и IP;
- серверная проверка всех входных данных;
- защита от массовой накрутки результатов.

Так как система работает с пользовательской активностью, важно, чтобы клиент не мог сам “решить”, сколько у него голосов — это должно определяться только сервером.

---

### 8. Наблюдаемость

Система должна быть достаточно прозрачной для сопровождения и отладки.

| Что отслеживаем | Минимальный набор |
|---|---|
| Логи | голосования, запросы на песни, ошибки API |
| Метрики | RPS, latency, количество активных подключений |
| Состояние realtime | число WebSocket/SSE-соединений, задержка доставки событий |
| База данных | нагрузка на чтение и запись |

Это нужно, чтобы можно было быстро понять:
- где появляется задержка;
- не теряются ли голоса;
- не падает ли realtime-доставка;
- как ведет себя система в часы пикового спроса.

---

### 9. Совместимость и пользовательская среда

| Показатель | Требование |
|---|---|
| Устройства | мобильные телефоны в приоритете |
| ОС | Android и iOS |
| Браузеры | современные мобильные браузеры |
| Сеть | работа при нестабильном соединении |

Поскольку основной сценарий — это быстрые действия со смартфона, интерфейс должен:
- корректно работать на маленьком экране;
- переживать кратковременные обрывы сети;
- повторно отправлять запросы, если соединение временно потеряно;
- не требовать сложной установки приложения для MVP.

---

### 10. Влияние на архитектуру

Из нефункциональных требований следуют основные архитектурные решения:
- realtime-обновления через WebSocket или SSE;
- stateless backend для горизонтального масштабирования;
- отдельное постоянное хранилище для голосов и запросов;
- кэш для часто читаемых данных;
- защита от повторных действий и спама;
- мониторинг задержек, ошибок и нагрузки.

## 1.5 Ограничения

## 1.5 Ограничения

### 1. Законодательство и регуляторика (РФ, 2026)

Система обрабатывает пользовательскую активность (голоса, запросы песен, идентификаторы пользователей), поэтому необходимо учитывать требования российского законодательства к обработке персональных данных.


- Использование зарубежных облачных провайдеров (AWS, GCP, Azure) как основной инфраструктуры нежелательно из-за рисков доступности и ограничений.
- Предпочтение отдается российским облачным провайдерам и self-hosted/open-source решениям.
- Система не хранит музыкальные файлы и не занимается лицензированием контента — ответственность за музыкальное вещание остается на стороне радиостанции.

---

### 2. Бизнесовые ограничения

Система проектируется как MVP для локальной радиостанции.

- Основная цель — повысить вовлеченность аудитории, а не построить полноценную стриминговую платформу.
- Монетизация, реклама, подписки и платные функции не входят в первую версию продукта.
- Ожидается локальная аудитория одного города или региона, а не глобальная распределенная нагрузка.
- Основной пользовательский сценарий — быстрое взаимодействие с эфиром во время прослушивания радио.

Это позволяет отказаться от избыточно сложной архитектуры уровня крупных медиаплатформ.

---


### 3. Технологические ограничения

Система должна поддерживать near-real-time обновления состояния эфира и результатов голосования.

Это накладывает ограничения на архитектуру:
- polling не должен быть основным механизмом синхронизации;
- требуется поддержка WebSocket или SSE;
- backend должен быть stateless для горизонтального масштабирования;
- realtime-соединения должны корректно переживать переподключения.

Дополнительно:
- система должна быть read-heavy optimized;
- hot data (текущий трек, счетчики голосов) желательно хранить в кэше;
- persistent storage обязательно для голосов и запросов.

---

### 4. Интеграции и внешние зависимости

Система не заменяет существующую радио-инфраструктуру, а работает поверх нее.

Внешние зависимости:
- источник метаданных текущего эфира;
- DJ-панель или система управления плейлистом;
- сервис авторизации (опционально в будущем).

На MVP:
- сложные внешние интеграции отсутствуют;
- интеграции с социальными сетями не обязательны;
- AI-сервисы и рекомендательные системы не используются.

---

### 5. Ограничения пользовательской платформы

Основной сценарий использования — мобильные устройства.

Требования:
- корректная работа в мобильных браузерах;
- адаптивный интерфейс;
- минимальное количество действий для голосования;
- устойчивость к нестабильному мобильному интернету.

Отдельное мобильное приложение не является обязательным для MVP.

---
## 1.6 Расчет нагрузки

Расчет сделан для локальной радиостанции с мобильной аудиторией.  
Система ориентирована не на глобальный масштаб, а на аудиторию одного города или региона, поэтому используем оценки для MVP и закладываем запас на пиковую активность во время эфира.

---

### 1. Исходные предположения

| Параметр | Значение |
|---|---|
| Monthly Active Users (MAU) | 20 000 |
| Daily Active Users (DAU) | 5 000 |
| Одновременные пользователи в пике | 2 000 |
| Доля пользователей, совершающих активное действие в день | 20–30% |
| Среднее число действий на активного пользователя в день | 3–6 |
| Доля трафика, приходящаяся на realtime-обновления | высокая |

Допущение: большинство пользователей не голосуют постоянно, а в основном открывают приложение, смотрят текущий трек и иногда голосуют или отправляют запрос.

---

### 2. Типы пользовательской активности

Для этой системы важны три основных класса нагрузки:

1. **Чтение состояния эфира**  
   Пользователь открывает приложение и получает текущий трек, плейлист и состояние голосования.

2. **Запись пользовательских действий**  
   Голосование за трек, отправка запроса на песню, отправка реакции на плейлист.

3. **Realtime-доставка обновлений**  
   Система должна почти сразу раздавать изменения всем активным клиентам: смена трека, новые голоса, новые запросы.

Поэтому нагрузка здесь не только “API-запросы”, но и количество активных соединений, через которые идет доставка событий.

---

### 3. Оценка пользовательских запросов

Примем консервативное приближение:

| Действие | Среднее количество на одного DAU в день |
|---|---|
| Открытие приложения / получение состояния эфира | 3 |
| Голосование | 1 |
| Отправка запроса на песню | 0.2 |
| Просмотр/обновление плейлиста | 2 |

Тогда суточная нагрузка составит:

- открытие приложения и чтение состояния:  
  `5 000 × 3 = 15 000` запросов/день

- голосование:  
  `5 000 × 1 = 5 000` запросов/день

- запросы песен:  
  `5 000 × 0.2 = 1 000` запросов/день

- обновление/просмотр плейлиста:  
  `5 000 × 2 = 10 000` запросов/день

**Итого:** около `31 000` прикладных операций в день.

Средний RPS:

`31 000 / 86 400 ≈ 0.36 req/s`

Это среднее значение очень небольшое, но для этой системы оно не показательно, потому что реальная нагрузка концентрируется в часы эфира.

---

### 4. Пиковая нагрузка

Для локальной радиостанции пики обычно возникают:
- вечером;
- во время популярных программ;
- при объявлении голосования за трек;
- сразу после смены песни.

Примем, что в пиковый час приходится около **20% суточной активности**.

`31 000 × 0.20 = 6 200` операций в час

Тогда пиковый RPS:

`6 200 / 3 600 ≈ 1.7 req/s`

С учетом кратковременных всплесков, повторных попыток и того, что несколько действий происходят одновременно, проектный пик можно оценить как:

**5–10 req/s по HTTP API**

Это уже реалистичный ориентир для небольшого продакшн-сервиса с запасом.

---

### 5. Realtime-нагрузка

Для этой задачи важно не только число HTTP-запросов, но и количество активных клиентов, которым нужно синхронно раздавать обновления.

Если в пике онлайн находится около `2 000` пользователей, и хотя бы `1 500` из них поддерживают активное realtime-соединение, то система должна уметь:
- держать много одновременных WebSocket/SSE-соединений;
- быстро рассылать события о смене трека;
- обновлять счетчики голосов без полного обновления страницы.

Если, например, в момент смены трека событие нужно доставить `1 500` клиентам, то это уже не обычный request/response сценарий, а **fan-out**-нагрузка.

Практический вывод:
- API по голосованию может быть легким;
- realtime-слой должен быть рассчитан на большое количество долгоживущих соединений;
- основной риск — не CPU на вычислениях, а количество одновременных соединений и скорость доставки событий.

---

### 6. Оценка размера данных

Основные сущности системы:

| Сущность | Оценка размера |
|---|---|
| Профиль пользователя | ~1 КБ |
| Песня / трек | ~0.5 КБ |
| Голос | ~150–200 байт |
| Запрос на песню | ~300–500 байт |
| Событие обновления эфира | ~100 байт |

#### Голоса
Если в день создается около `5 000` голосов:

`5 000 × 200 B = 1 000 000 B ≈ 1 MB/day`

За год:

`1 MB × 365 ≈ 365 MB/year`

#### Запросы песен
Если в день около `1 000` запросов:

`1 000 × 400 B = 400 000 B ≈ 0.4 MB/day`

За год:

`0.4 MB × 365 ≈ 146 MB/year`

#### История эфира и служебные события
Допустим, в день создается несколько тысяч событий, но они небольшие по размеру. Даже при хранении истории за длительный период общий объем останется умеренным.

**Вывод:** по хранению система не является тяжелой.  
Основная нагрузка — не на объем данных, а на скорость обновлений и одновременную доставку информации пользователям.

---

### 7. Нагрузка на базу данных

Для базы данных важны:
- запись голосов и запросов;
- чтение текущего состояния эфира;
- чтение истории голосования и плейлиста.

Так как чтений больше, чем записей, нагрузка будет **read-heavy**.

Оценочно:

| Тип операции | Доля |
|---|---|
| Чтение текущего состояния | 60–70% |
| Чтение плейлиста и результатов | 20–25% |
| Запись голосов и запросов | 10–20% |

Даже одна PostgreSQL-инстанция справится с такой нагрузкой на уровне MVP, но для отказоустойчивости и роста лучше сразу предусмотреть:
- primary + standby;
- кэш для текущего состояния;
- отдельное хранение realtime-событий или публикацию через очередь / pub-sub.

---

### 8. Сетевой трафик

Если средний ответ API на чтение состояния составляет `2–5 КБ`, а пик чтений — около `5–10 req/s`, то входящий и исходящий HTTP-трафик остается небольшим.

Пример:
- `10 req/s × 5 КБ = 50 КБ/с`

Это очень скромная нагрузка для сети.

Но realtime-соединения создают постоянный фоновый трафик:
- периодические heartbeat-сообщения;
- доставка событий о смене трека;
- обновление счетчиков голосов.

Поэтому сеть должна быть рассчитана не только на объем данных, но и на большое число долгоживущих соединений.

---

### 9. Оценка вычислительных ресурсов

Для MVP достаточно следующей стартовой конфигурации:

| Компонент | Минимальная конфигурация | Назначение |
|---|---|---|
| Backend API | 2 vCPU, 2–4 GB RAM, 2 реплики | Обработка запросов пользователей |
| Realtime-сервис | 2 vCPU, 2 GB RAM, 2 реплики | WebSocket / SSE и доставка событий |
| База данных | 2–4 vCPU, 4–8 GB RAM | Хранение голосов, запросов, плейлистов |
| Кэш | небольшой Redis-инстанс | Текущее состояние эфира и горячие данные |
