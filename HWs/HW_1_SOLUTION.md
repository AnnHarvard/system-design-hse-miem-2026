# HW 1. Нагрузочное тестирование demo-app-1

## 1. Архитектура системы

`demo-app-1` — учебное веб-приложение для работы с пользователями и заказами. Вся система поднимается через Docker Compose и включает полный стек observability.

### Компоненты системы
- **Nginx** — reverse proxy и точка входа для UI и API.
- **Backend (Go)** — REST API для пользователей и заказов.
- **PostgreSQL** — основное хранилище данных.
- **Prometheus** — сбор метрик.
- **Grafana** — визуализация метрик.
- **node-exporter**, **postgres-exporter**, **cAdvisor** — инфраструктурные метрики.
- **k6** — генерация нагрузки.

### Схема взаимодействия

`Client -> Nginx -> Backend -> PostgreSQL`

Prometheus собирает метрики с backend, PostgreSQL, node-exporter и cAdvisor, а Grafana отображает их на дашбордах.

---

## 2. Какие метрики важно отслеживать

Для анализа системы под нагрузкой были выбраны следующие метрики.

### Метрики приложения
- RPS (Requests Per Second)
- количество запросов по кодам ответа (`2xx`, `4xx`, `5xx`)
- latency: `p50 / p95 / p99`
- количество таймаутов

### Метрики backend
- CPU usage
- RAM usage
- количество ошибок на уровне HTTP
- время выполнения SQL-запросов

### Метрики PostgreSQL
- QPS
- количество активных соединений
- число чтений и записей
- наличие conflicts / deadlocks
- время ответа БД

### Метрики хоста и контейнеров
- CPU Busy
- Sys Load
- RAM Used
- I/O pressure
- network / disk usage

## 3. Сценарии нагрузочного тестирования

### Сценарий 1. «Шторм»
Резкий пик нагрузки: быстрый рост числа виртуальных пользователей до **1000** за короткое время.

**Цель:** проверить, как система реагирует на внезапный burst и где появляется первая деградация.

### Сценарий 2. «Волна»
Плавный рост нагрузки от **0 до 500 пользователей** за **2 минуты**.

**Цель:** посмотреть, в какой момент система начинает деградировать при постепенном увеличении нагрузки.

### Сценарий 3. Модификация сценария
В качестве дополнительной проверки использовался сценарий с перекосом в сторону чтения.

Пример:
- **read-heavy**: 90% `GET /api/orders`, 10% `POST /api/orders`

**Фишка сценария:** проверить, влияет ли уменьшение доли записей на latency и процент ошибок.

### Примечание по k6
Метрики k6 не удалось стабильно вывести в Prometheus / Grafana, поэтому для анализа использовались:
- консольный вывод k6;
- дашборды backend / PostgreSQL / node-exporter;
- запросы в Prometheus.

## 4. Результаты тестирования

### 4.1. Сценарий «Шторм»

#### Параметры
Использовался сценарий с резким ростом нагрузки до **1000 VU** и распределением запросов:
- 80% `POST /api/orders`
- 20% `GET /api/orders`

#### Итоги k6
- total requests: **298 926**
- request rate: **1245.27 req/s**
- checks succeeded: **47.56%**
- checks failed: **52.43%**
- `http_req_failed`: **52.43%**
- `http_req_duration`:
  - avg: **126.92 ms**
  - med: **6.35 ms**
  - p(90): **310.81 ms**
  - p(95): **588.32 ms**
  - max: **6.52 s**
- `vus_max`: **1000**

#### Проверки по типам запросов
- `created` (`POST /api/orders`): **39%** success
- `list ok` (`GET /api/orders`): **79%** success

#### Дополнительные наблюдения
- CPU Pressure: **97.8%**
- CPU Busy: **17.5%**
- Sys Load: **442.4%**
- RAM Used: **22.6%**
- I/O Pressure: **1.6%**
- PostgreSQL QPS: **306**
- `POST /api/orders 200`: **81.75 req/s**
- `POST /api/orders 500`: **82.12 req/s**
- `GET /api/orders 200`: **38.32 req/s**

#### Вывод
При пиковом сценарии система начала массово возвращать ошибки `500` на `POST /api/orders`. Это указывает на bottleneck в цепочке `backend -> PostgreSQL` при write-heavy нагрузке.

## 5. Анализ и выводы

### Что видно по системе
1. Нагрузка была высокой: k6 сгенерировал сотни тысяч запросов с пиками до **1000 VU**.
2. Система начала заметно деградировать по ошибкам: доля неуспешных запросов доходила до **52–74%**.
3. Особенно уязвимы операции записи: `POST /api/orders` давал большое число `500`.
4. По CPU и памяти сервер не выглядит полностью исчерпанным: RAM держится примерно на уровне **18–23%**, CPU Busy — от **17% до 58%** в зависимости от сценария.
5. PostgreSQL выдерживает существенный поток запросов, но именно цепочка `backend -> PostgreSQL` при write-heavy нагрузке выглядит наиболее проблемной.

### Предварительный bottleneck
Основной кандидат на bottleneck — **обработка записей в PostgreSQL через backend**, а также связанный с этим пул соединений и синхронная обработка запросов.

Система начинает отдавать ошибки раньше, чем полностью упирается в память или диск. Это указывает на ограничение в области обработки write-запросов, транзакций или SQL-операций.

### Ключевой инсайт
Длительная нагрузка опаснее краткого всплеска. В сценарии «Волна» показатели оказались хуже, чем в сценарии «Шторм»: это означает, что система чувствительна не только к burst-нагрузке, но и к sustained-нагрузке, когда ресурсы долго находятся под давлением.

---

## 6. Выявленные узкие места

По результатам текущих прогонов можно сделать такие выводы:

- память не является главным ограничением системы;
- диск не выглядит проблемным местом;
- PostgreSQL обрабатывает высокий поток запросов, но основная нагрузка идёт на операции чтения и особенно записи;
- на backend появляются ошибки `500` для `POST /api/orders`;
- вероятный bottleneck — цепочка `backend -> PostgreSQL` при большом числе write-запросов;
- при длительной нагрузке система деградирует сильнее, чем при коротком пике.



---

## 7. Предложения по улучшению

Если система должна выдерживать больший поток запросов, можно рассмотреть следующие меры:

- оптимизировать SQL-запросы и добавить индексы;
- увеличить и настроить пул соединений к PostgreSQL;
- уменьшить число синхронных операций в backend;
- добавить кэширование для чтения;
- вынести чтение на реплики БД;
- использовать очередь сообщений для асинхронной обработки части write-операций;
- добавить горизонтальное масштабирование backend через Nginx / HAProxy;
- добавить rate limiting или backpressure при burst-нагрузке.
