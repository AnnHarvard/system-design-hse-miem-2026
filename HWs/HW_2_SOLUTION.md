# HW 2. Patroni PostgreSQL 

## 1. Цель работы

Цель домашнего задания — посмотреть, как работает отказоустойчивый кластер PostgreSQL на базе Patroni, etcd и HAProxy, а также проверить поведение системы при отказах отдельных компонентов.

В работе проверялось:
- состав кластера и роли узлов;
- маршрутизация через HAProxy;
- запись и чтение данных через мастер и реплики;
- автоматический failover при отказе лидера;
- поведение кластера при потере etcd;
- влияние отказа HAProxy на доступность приложения.

---

## 2. Архитектура решения

Используется стек:

- **Patroni** — управляет PostgreSQL-узлами, выбирает лидера и следит за состоянием кластера.
- **etcd** — DCS (Distributed Configuration Store), хранит состояние кластера и лидерский ключ.
- **HAProxy** — принимает запросы от приложения и перенаправляет их на актуальный узел.
- **PostgreSQL** — хранение данных.
- **traffic-generator.py** — скрипт для проверки записи и чтения данных.

Схема работы:

`client / traffic-generator -> HAProxy -> Patroni cluster -> PostgreSQL`

Patroni использует etcd для координации выбора лидера. HAProxy опрашивает Patroni и отправляет трафик на актуальную master-ноду.

Одна нода является лидером и принимает записи, две другие работают как реплики и синхронизируются с лидером в режиме streaming replication.

## 8. Эксперименты с отказоустойчивостью

### 8.1. Отключение одной реплики

Был остановлен контейнер:

```bash
docker stop demo-patroni1
```

После этого `patronictl list` показал:
- `patroni1` — `stopped`
- `patroni2` — `Replica`
- `patroni3` — `Leader`

### Наблюдение
traffic-generator продолжил работу без ошибок. Записи и чтение данных не прерывались.

### Вывод
Отказ одной реплики не влияет на доступность системы. Лидер продолжает обслуживать запись, оставшаяся реплика продолжает синхронизацию.

---

### 8.2. Отключение лидера

После этого был остановлен лидер кластера.

Во время переключения traffic-generator показал ошибку вида:

```text
CONNECTION LOST (Failover in progress?)
```

Затем выполнение восстановилось.  
Позже `patronictl list` показал:

- `patroni2` — **Leader**
- `TL = 2`

### Наблюдение
Patroni автоматически провел failover, и новый лидер был выбран без ручного вмешательства.

### Вывод
При отказе лидера происходит кратковременная недоступность во время выбора нового мастера, но затем кластер восстанавливается автоматически. Это подтверждает работу механизма failover.

---

### 8.3. Отключение одной ноды etcd

Был остановлен контейнер:

```bash
docker stop demo-etcd1
```

После этого Patroni начал выводить предупреждения о недоступности etcd1, но кластер продолжил работу.

### Наблюдение
- лидер оставался доступен;
- чтение и запись продолжались;
- traffic-generator не потерял соединение.

### Вывод
Кластер etcd устойчив к отказу одной ноды, так как сохраняется кворум 2 из 3 узлов.

---

### 8.4. Отключение второй ноды etcd

После этого была остановлена еще одна нода etcd:

```bash
docker stop demo-etcd2
```

После потери двух узлов Patroni начал выдавать ошибки:

```text
EtcdConnectionFailed: No more machines in the cluster
Etcd3Error: Etcd is not responding properly
```

Команда `patronictl list` перестала работать, так как состояние кластера больше не могло быть получено из DCS.

### Вывод
При потере кворума etcd Patroni теряет возможность координации кластера.  
Это ожидаемое поведение для Raft-based DCS: при недоступности большинства узлов кворум теряется.

---

### 8.5. Отключение HAProxy

Был остановлен контейнер:

```bash
docker stop demo-haproxy
```

После этого traffic-generator начал выдавать:

```text
Connection refused
```

на порту `5002`.

### Вывод
PostgreSQL-кластер и Patroni продолжали существовать, но приложение потеряло точку входа в систему.  
Это означает, что HAProxy в данной конфигурации является Single Point of Failure.

## 9. Итоговые выводы

В ходе работы было проверено:

- как Patroni выбирает лидера;
- как работает streaming replication;
- как HAProxy направляет запросы к актуальному master;
- как происходит failover при отказе лидера;
- что происходит при потере отдельных нод etcd;
- как влияет отказ HAProxy на доступность приложения.

### Основные выводы

1. Кластер PostgreSQL с Patroni действительно повышает отказоустойчивость.
2. Отключение реплики не влияет на работу системы.
3. При отказе лидера Patroni автоматически выбирает нового лидера.
4. etcd устойчив к отказу одной ноды, но потеря кворума приводит к ошибкам координации.
5. HAProxy является единой точкой входа и без резервирования остается SPOF.
6. Для продакшена HAProxy нужно резервировать: использовать второй балансировщик, VIP/Keepalived или внешний load balancer.

---

## 10. Что можно улучшить в продакшене

Для повышения надежности и доступности решения можно:

- добавить второй HAProxy;
- использовать Keepalived или виртуальный IP;
- следить за quorum etcd и количеством активных узлов;
- настроить мониторинг и алерты на состояние Patroni / etcd / HAProxy;
- вынести клиентский доступ на более отказоустойчивый слой балансировки.
